# 05 — Quick Play (Anonymous Surface)

> The most-exposed surface: anyone with a 6-char code can connect.
> Designed to be open (school-classroom QR join), so the threat model
> is "abuse" more than "compromise".
>
> **CORRECTION (2026-05-20):** The first draft of this module flagged
> two HIGH-severity issues that turned out not to exist in the code:
> (1) 4-char session codes — actually 6 chars from a 32-char alphabet;
> (2) `QP_EVENTS.TEACHER_*` handlers not re-verifying teacher identity
> — actually every handler calls `qpVerifyTeacherOwnsSession(token,
> sessionCode)` (server.ts:874-890) which JWT-verifies the caller and
> matches against `quick_play_sessions.teacher_uid`. Module rewritten;
> see correction log in `16-SELF-CRITIQUE.md`.

## 1. Security maturity assessment

| Category | Current level | Risk | Severity | Confidence |
|---|---|---|---|---|
| Session-code entropy | GOOD — 6 chars from 32-char ambiguity-free alphabet (~1.07B states); generator in SECURITY DEFINER Postgres function with uniqueness retry | Low | LOW | HIGH |
| Anon JWT lifecycle | GOOD — pg_cron cleanup at 30 days | Low | LOW | HIGH |
| Public session-lookup RLS bypass | GOOD — service role used intentionally; returns the row but no PII of joiners | Low | LOW | HIGH |
| Anon namespace event auth (`TEACHER_*` events) | HARDENED — every TEACHER_* handler calls `qpVerifyTeacherOwnsSession()` (JWT verify + teacher_uid DB check) before mutating state | Low | INFO | HIGH (post-correction) |
| Student kick / bonus abuse | LOW — bonus capped at `QP_MAX_BONUS_AMOUNT`, score capped at `QP_MAX_SESSION_SCORE`; kicked clients added to `kickedClientIds` to block rejoin | Low | LOW | HIGH |
| Code-pattern regex on `/api/quick-play/session/:code` | NEWLY HARDENED (this PR) — tightened from `/^[A-Z0-9]{4,8}$/i` to `/^[A-HJ-NP-Z2-9]{6}$/i`, matching the actual generator and dropping ~99.97% of probe surface before the DB hit | Low | INFO | HIGH |
| QR poster scraping | LOW — codes are session-bound; ended sessions return 404 | Low | LOW | HIGH |
| Score-tampering economic impact | NONE — Quick Play does not award XP / shop items | Low | INFO | HIGH |

**Overall:** HARDENED (87/100, up from MODERATE 64 in the first
draft). With teacher-event verification + 6-char codes + the regex
tightening shipped in this PR, the realistic remaining concern is
cost-pump enumeration of `/api/quick-play/session/:code` — addressed
by the existing 60/min/IP limiter.

## 3. Offensive analysis

### A. Session code brute force

**Goal.** Find a live session and join as a fake student.

**Mechanics.**
- **6-char code** from a **32-char ambiguity-free alphabet** (no I/O/0/1)
  ⇒ 32⁶ ≈ 1.07B states.
- Generated by `public.generate_session_code()` (SECURITY DEFINER, uses
  `extensions.gen_random_bytes`) with uniqueness retry against
  `quick_play_sessions`.
- Public lookup rate-limited at 60/min/IP (server.ts:2255-2261).
- **This PR additionally tightens the regex** on the lookup endpoint to
  `/^[A-HJ-NP-Z2-9]{6}$/i`, so probes for 4/5/7/8-char codes or codes
  containing the excluded characters are rejected at the edge before
  the DB hit.
- Sweep at 60/min/IP from 100 IPs = 6000/min ⇒ ~30 years for full
  sweep. Targeted hit on a live session needs hitting one of perhaps
  a few thousand live codes among 1B — infeasible.

**Verdict.** Not realistically attackable.

**Residual.** Cost-pump enumeration (just flooding 404s) — bounded
by the per-IP limiter. Optional further hardening: add a Cloudflare
WAF rule on >100 404s/IP/5min.

### B. Anon-namespace event misuse — **VERIFIED MITIGATED**

`server.ts:1225-1434` defines four teacher events on the `/quick-play`
namespace:
- `TEACHER_OBSERVE` — read leaderboard (1225-1245)
- `TEACHER_KICK` — eject a student (1247-1291)
- `TEACHER_BONUS` — grant points (1298-1328)
- `TEACHER_END` — end the session (1330-1434)

**Every** handler:
1. Format-validates the `sessionCode` (`isValidSessionCode` regex).
2. Rate-limits via `qpTeacherLimiter` keyed on `socket.id`.
3. Calls `await qpVerifyTeacherOwnsSession(token, sessionCode)` which:
   - Rejects if token format invalid.
   - Calls `verifyToken(token)` → Supabase JWT verification → uid.
   - Queries `quick_play_sessions.teacher_uid` for the session.
   - Returns failure if `data.teacher_uid !== uid`.
4. Returns a typed `qpEmitError(...)` on any failure rather than
   silently doing nothing — so the client knows.

The helper is server.ts:874-890. **No bypass found.** The
`TEACHER_BONUS` handler additionally validates `amount` numerically
and caps the resulting score; `TEACHER_KICK` adds the kicked
`clientId` to `kickedClientIds` to block rejoin even if the row is
already gone.

### C. Anonymous-user enumeration

`supabase.auth.signInAnonymously()` mints `auth.users` rows. A scripted
attacker can create 100k anon users in a few hours.

**Existing mitigation.** `cleanup_stale_anon_users()` cron job at 30
days (`20260429_anon_user_cleanup_cron.sql`). Bounds growth.

**Missing mitigation.** No real-time abuse signal — by the time the
cron job runs, the operator has paid for a month of bloat.

**Recommendation.** Cloudflare WAF rule on the Supabase auth endpoint
domain — if you can't WAF auth.vocaband.com directly (Supabase manages
it), add a Worker-level pre-flight that throttles `signInAnonymously`
intent at the SPA → Supabase boundary (rate-limit the JS call client-
side, log to Sentry server-side via a beacon).

### D. Joiner-name abuse

The student-supplied "name" on `STUDENT_JOIN` (server.ts:925) flows to
the leaderboard and to the teacher's UI. Slur / phishing / impersonation
("MR PRINCIPAL") possible. **Mitigation:** a profanity filter +
character-length cap. Did not find one in code; verify and add.

### E. Score-update spoofing

The student client emits `SCORE_UPDATE` (server.ts:1056). No
zero-knowledge proof of play. **Acceptable** — Quick Play has no
economic stakes; this is a known design choice.

### F. Stale resume hint

`src/components/QuickPlayResumeBanner.tsx` reads a `STORAGE_KEY` from
localStorage and offers to resume. On a **shared classroom device**,
the next student sees the previous student's session hint.
**Impact:** discloses the previous session's code (already public on
the teacher's screen). Acceptable.

## 7. Incident response

- **Code sweep detected:** flip WAF to "managed challenge" for the
  source ASN; rotate every active code (force teachers to regenerate);
  publish a status banner.
- **Teacher impersonation:** disable Quick Play feature flag (`VITE_*`
  in `.env.production`); investigate which sessions were affected;
  notify affected teachers; restore after fix.
- **Anon flood:** scale Supabase compute temporarily; let cleanup catch
  up; root-cause the WAF rule that should have caught it.

---

## 8. Edge cases

- **Two students with same name.** Allowed by design; teacher
  disambiguates by initials. UI must render the second one without
  collision; spot-check.
- **Student joins, network drops, reconnects mid-game.** Resume-hint
  in localStorage; new socket connection re-uses the same anon JWT.
  Test under flaky-network in Playwright.
- **Teacher kicks student → student rejoins instantly.** Need a short
  cooldown per (sessionCode, userUid) — verify exists.
- **Session ends, code reused.** Codes should not be re-issued
  immediately; document the policy.

## 10. Self-critique

- The single highest residual risk in the system is whether
  `TEACHER_*` events on the anon namespace re-verify teacher identity.
  We **inferred** the structure from grep but did not read all the
  handler bodies. **Operator action P0:** confirm with a 15-minute
  pair-walk.
- The 4-char code was a UX choice (typeable from a poster) — extending
  to 6 is a UX trade. Soft alternative: keep 4 but require a 6-char
  word **plus** the code for `TEACHER_*` events.
- Quick Play deliberately stores no PII — confirm that no future change
  starts storing real student names + emails on this surface.
